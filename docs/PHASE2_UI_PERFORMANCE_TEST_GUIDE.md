## 반응형 디자인 테스트

### 테스트 항목: 다양한 디바이스에서 레이아웃 확인

#### 1. 모바일 뷰 테스트 (320px ~ 640px)

**절차:**
1. 브라우저 개발자 도구에서 모바일 뷰 활성화
2. 다양한 모바일 기기 크기로 테스트:
   - iPhone SE (375px)
   - iPhone 12/13 (390px)
   - Samsung Galaxy (360px)
3. 각 페이지별 확인:
   - 메인 페이지 (`/`)
   - 퇴사자 페이지 (`/resigned`)
   - 대시보드 (`/dashboard`)
   - 알림 페이지 (`/notifications`)

**예상 결과:**
- ✅ 네비게이션 햄버거 메뉴가 표시됨
- ✅ 직원 카드가 1열로 표시됨
- ✅ 페이지네이션 버튼이 터치하기 쉬운 크기 (최소 44x44px)
- ✅ EmployeeForm이 1열 레이아웃으로 표시됨
- ✅ 탭이 가로 스크롤 가능
- ✅ 모든 버튼이 터치하기 쉬운 크기
- ✅ 텍스트가 읽기 쉬운 크기 (최소 14px)

#### 2. 태블릿 뷰 테스트 (641px ~ 1024px)

**절차:**
1. 브라우저 개발자 도구에서 태블릿 뷰 활성화
2. 다양한 태블릿 크기로 테스트:
   - iPad (768px)
   - iPad Pro (1024px)
3. 각 페이지별 확인

**예상 결과:**
- ✅ 직원 카드가 2열로 표시됨
- ✅ 네비게이션이 전체 메뉴로 표시됨
- ✅ 레이아웃이 모바일과 데스크톱 사이의 적절한 중간 형태

#### 3. 데스크톱 뷰 테스트 (1025px 이상)

**절차:**
1. 브라우저를 전체 화면으로 확대
2. 다양한 해상도로 테스트:
   - 1280px (HD)
   - 1920px (Full HD)
   - 2560px (2K)
3. 각 페이지별 확인

**예상 결과:**
- ✅ 직원 카드가 3열로 표시됨
- ✅ 네비게이션이 전체 메뉴로 표시됨
- ✅ 레이아웃이 최대 너비를 효율적으로 사용
- ✅ 텍스트와 요소들이 적절한 간격 유지

#### 4. 반응형 레이아웃 전환 테스트

**절차:**
1. 브라우저 창 크기를 점진적으로 조절
2. 각 브레이크포인트에서 레이아웃 변화 확인:
   - 640px (모바일 → 태블릿)
   - 1024px (태블릿 → 데스크톱)

**예상 결과:**
- ✅ 레이아웃이 부드럽게 전환됨
- ✅ 콘텐츠가 잘리지 않음
- ✅ 스크롤이 정상 작동

## 다크 모드 테스트

### 테스트 항목: 다크 모드 전환 및 새로고침 후 유지 확인

#### 1. 다크 모드 전환 테스트

**절차:**
1. 네비게이션에서 테마 전환 버튼 클릭
2. 라이트 모드 → 다크 모드 전환 확인
3. 다크 모드 → 라이트 모드 전환 확인

**예상 결과:**
- ✅ 테마가 즉시 전환됨
- ✅ 모든 페이지에서 일관된 테마 적용
- ✅ 색상 대비가 적절함 (WCAG AA 기준)
- ✅ 텍스트가 읽기 쉬움
- ✅ 이미지와 아이콘이 적절히 표시됨

#### 2. 새로고침 후 테마 유지 테스트

**절차:**
1. 다크 모드로 설정
2. 브라우저 새로고침 (F5)
3. 테마 상태 확인

**예상 결과:**
- ✅ 다크 모드가 유지됨
- ✅ localStorage에 테마 설정이 저장됨
- ✅ 페이지 로드 시 저장된 테마가 자동 적용됨

#### 3. 시스템 테마 연동 테스트

**절차:**
1. 브라우저 설정에서 시스템 테마 변경
2. 앱의 테마 상태 확인

**예상 결과:**
- ✅ 시스템 테마 변경 시 앱 테마도 변경됨 (next-themes 기본 동작)
- ✅ 또는 사용자가 설정한 테마가 우선 적용됨

#### 4. 다크 모드 색상 대비 확인

**절차:**
1. 다크 모드 활성화
2. 주요 텍스트와 배경의 색상 대비 확인
3. WCAG AA 기준 충족 여부 확인

**예상 결과:**
- ✅ 일반 텍스트: 최소 4.5:1 대비 비율
- ✅ 큰 텍스트: 최소 3:1 대비 비율
- ✅ 버튼 텍스트: 최소 4.5:1 대비 비율
- ✅ 포커스 표시가 명확함

## 애니메이션 테스트

### 테스트 항목: 애니메이션 부드러움 확인

#### 1. 직원 카드 등장 애니메이션

**절차:**
1. 메인 페이지 접속
2. 직원 카드가 로드되는 과정 관찰
3. 페이지 스크롤 시 추가 카드 로드 관찰

**예상 결과:**
- ✅ 카드가 순차적으로 부드럽게 등장 (fadeIn, slideUp)
- ✅ 애니메이션 지속 시간이 적절함 (0.3s ~ 0.5s)
- ✅ 애니메이션이 60fps로 부드럽게 실행됨
- ✅ 성능 저하 없음

#### 2. 모달 애니메이션

**절차:**
1. 직원 추가/수정 버튼 클릭
2. 모달 등장 애니메이션 확인
3. 모달 닫기 버튼 클릭
4. 모달 퇴장 애니메이션 확인

**예상 결과:**
- ✅ 모달이 부드럽게 등장 (scaleIn, fadeIn)
- ✅ 모달이 부드럽게 퇴장 (scaleOut, fadeOut)
- ✅ 배경 오버레이도 함께 애니메이션
- ✅ 애니메이션이 자연스러움

#### 3. 호버 효과

**절차:**
1. 직원 카드에 마우스 호버
2. 버튼에 마우스 호버
3. 호버 효과 확인

**예상 결과:**
- ✅ 카드가 부드럽게 상승 및 확대 (transform, box-shadow)
- ✅ 버튼이 부드럽게 확대 (scale)
- ✅ 전환 효과가 즉각적이지만 부드러움 (transition)
- ✅ 호버 해제 시 원래 상태로 부드럽게 복귀

#### 4. 로딩 애니메이션

**절차:**
1. 페이지 로드 시 로딩 스피너 확인
2. 데이터 로드 중 스켈레톤 UI 확인

**예상 결과:**
- ✅ 로딩 스피너가 부드럽게 회전
- ✅ 스켈레톤 UI가 순차적으로 등장
- ✅ 펄스 효과가 자연스러움
- ✅ 로딩 상태가 명확히 표시됨

#### 5. 성능 확인

**절차:**
1. 브라우저 개발자 도구의 Performance 탭 열기
2. 페이지 로드 및 인터랙션 기록
3. FPS 확인

**예상 결과:**
- ✅ FPS가 60fps 유지
- ✅ 메인 스레드 블로킹 없음
- ✅ 애니메이션이 GPU 가속 활용 (transform, opacity)

## 접근성 테스트

### 테스트 항목: 키보드만으로 모든 기능 사용 가능 확인

#### 1. 키보드 네비게이션 기본 테스트

**절차:**
1. Tab 키로 포커스 이동
2. Shift+Tab으로 역방향 이동
3. Enter/Space로 버튼 활성화
4. Esc로 모달 닫기

**예상 결과:**
- ✅ 모든 인터랙티브 요소에 포커스 가능
- ✅ 포커스 순서가 논리적 (위에서 아래, 왼쪽에서 오른쪽)
- ✅ 포커스 표시가 명확함 (ring-2 스타일)
- ✅ 키보드로 모든 기능 사용 가능

#### 2. 스킵 링크 테스트

**절차:**
1. 페이지 로드 후 Tab 키 누르기
2. 스킵 링크 확인

**예상 결과:**
- ✅ "메인 콘텐츠로 건너뛰기" 링크가 첫 번째로 포커스됨
- ✅ Enter 키로 메인 콘텐츠로 이동
- ✅ 네비게이션을 건너뛰고 콘텐츠로 바로 이동 가능

#### 3. 모달 키보드 네비게이션

**절차:**
1. Tab 키로 모달 내 요소 탐색
2. 모달 밖 요소로 포커스 이동 시도
3. Esc 키로 모달 닫기

**예상 결과:**
- ✅ 모달 내에서만 포커스 순환 (포커스 트랩)
- ✅ 모달 밖 요소로 포커스 이동 불가
- ✅ Esc 키로 모달 닫기 가능
- ✅ 모달 닫은 후 이전 포커스 위치로 복귀

#### 4. 폼 키보드 네비게이션

**절차:**
1. EmployeeForm 열기
2. Tab 키로 모든 입력 필드 탐색
3. Enter 키로 제출

**예상 결과:**
- ✅ 모든 입력 필드에 순차적으로 포커스 가능
- ✅ 라디오 버튼과 체크박스도 키보드로 선택 가능
- ✅ Enter 키로 폼 제출 가능
- ✅ 유효성 검증 오류 시 해당 필드로 포커스 이동

#### 5. 스크린 리더 테스트 (선택)

**절차:**
1. 스크린 리더 활성화 (NVDA, JAWS, VoiceOver 등)
2. 페이지 탐색
3. 주요 기능 확인

**예상 결과:**
- ✅ 모든 버튼에 적절한 aria-label 제공
- ✅ 이미지에 alt 텍스트 제공
- ✅ 폼 필드에 label 연결
- ✅ 현재 상태가 스크린 리더로 읽힘 (예: 활성 탭)
- ✅ 오류 메시지가 스크린 리더로 읽힘

## 성능 최적화 테스트

### 테스트 항목: 이미지 로딩 속도 측정

#### 1. 이미지 최적화 확인

**절차:**
1. 브라우저 개발자 도구의 Network 탭 열기
2. 페이지 로드
3. 이미지 로딩 시간 확인

**예상 결과:**
- ✅ `next/image` 컴포넌트 사용
- ✅ 이미지가 lazy loading으로 로드됨
- ✅ 이미지 크기가 적절함 (불필요하게 큰 이미지 없음)
- ✅ WebP 형식 사용 (선택사항)

#### 2. 이미지 크기 최적화 (압축)

**절차:**
1. 프로필 이미지 파일 크기 확인
2. 이미지 압축 도구 사용 (선택사항)

**예상 결과:**
- ✅ 이미지 파일 크기가 합리적 (예: 100KB 이하)
- ✅ 이미지 품질이 적절함
- ✅ 로딩 속도가 빠름

### 테스트 항목: React DevTools Profiler로 성능 측정

#### 1. 렌더링 성능 측정

**절차:**
1. React DevTools의 Profiler 탭 열기
2. "Start profiling" 클릭
3. 주요 인터랙션 수행 (직원 추가, 수정, 삭제)
4. "Stop profiling" 클릭
5. 성능 분석

**예상 결과:**
- ✅ 렌더링 시간이 합리적 (예: 16ms 이하)
- ✅ 불필요한 리렌더링 없음
- ✅ React.memo, useMemo, useCallback이 적절히 사용됨
- ✅ 컴포넌트 트리가 최적화됨

#### 2. 메모이제이션 확인

**절차:**
1. Profiler로 컴포넌트 렌더링 추적
2. 동일한 props로 리렌더링되는지 확인

**예상 결과:**
- ✅ React.memo가 적용된 컴포넌트는 props 변경 시에만 리렌더링
- ✅ useMemo로 계산된 값이 불필요하게 재계산되지 않음
- ✅ useCallback으로 메모이제이션된 함수가 불필요하게 재생성되지 않음
